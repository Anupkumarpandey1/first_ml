# HealthAI Diagnostics — Complete Architecture & Logic Documentation

## Overview

HealthAI Diagnostics is a full-stack AI-powered health report analysis platform. A user registers as a patient, uploads a lab report (image or PDF), and receives a structured medical analysis plus an interactive AI doctor chat. The system combines Gemini Vision, a LangGraph ReAct agent, RAG (Retrieval-Augmented Generation), and a classical ML risk model.

## Backend

### Framework: FastAPI

FastAPI serves as the HTTP layer. It handles CORS (open to all origins for dev), static file serving for `correlation.png`, and exposes these endpoints:

| Method | Endpoint | Purpose |
|--------|----------|---------|
| POST | `/patient/register` | Register patient, compute BMI, create session |
| POST | `/report/upload` | Upload lab report, extract data via Gemini Vision |
| POST | `/report/analyze` | Analyze extracted report with Gemini |
| POST | `/chat` | Send message to LangGraph ReAct agent |
| POST | `/patient/predict` | Legacy ML-based diabetes risk prediction |
| GET | `/static/correlation.png` | Serve the correlation heatmap image |

Sessions are stored in a plain Python dict (`_sessions`) keyed by UUID. Each session holds:
- `patient_info` — name, age, gender, height, weight, bmi
- `report_text` — raw extracted text/JSON from the report
- `report_analysis` — structured analysis dict from Gemini
- `chat_history` — list of `{role, content}` messages

---

### Report Extraction: Gemini Vision (`agent.py → extract_report_with_vision`)

**Library:** `google-genai` (Google GenAI Python SDK)  
**Model:** `gemini-2.5-flash`

Flow:
1. If the file is a `.pdf`, `pypdf.PdfReader` extracts raw text from all pages and sends it as a text prompt.
2. If the file is an image (`.png`, `.jpg`, `.jpeg`, `.webp`), it is base64-encoded and sent as an `inline_data` part.
3. The prompt instructs Gemini to:
   - Return `NOT_A_HEALTH_REPORT` if the file is not a medical document.
   - Otherwise, return a JSON object where each key is a test parameter name and the value contains `value`, `unit`, `reference_range`, and `status` (`normal/high/low`).
4. The backend checks for `NOT_A_HEALTH_REPORT` and raises HTTP 400 if found.
5. The JSON is parsed and stored in the session.

---

### Report Analysis: Gemini (`agent.py → analyze_report_with_gemini`)

**Library:** `google-genai`  
**Model:** `gemini-2.5-flash`

A detailed prompt is constructed combining the patient profile (name, age, gender, height, weight, BMI) and the extracted lab JSON. Gemini is asked to return a structured JSON with these keys:

- `summary` — 2-3 sentence plain-language overview
- `good_findings` — list of normal parameters with simple explanations
- `concerning_findings` — abnormal parameters with severity (`mild/moderate/serious`), explanation, and action advice
- `disease_risks` — potential diseases with `low/moderate/high` risk level
- `follow_up_tests` — recommended tests with urgency (`routine/soon/urgent`)
- `lifestyle_tips` — personalized actionable tips

The response is stripped of markdown fences and parsed as JSON. If parsing fails, the raw text is returned as the summary.

### RAG Pipeline (`rag_pipeline.py`)

**Libraries:** `langchain-community`, `langchain-text-splitters`, `faiss-cpu`, `sentence-transformers`

#### Building the Index (`build_rag_index`)
1. A `sample_medical_guidelines.txt` file is written with curated medical content covering diabetes, hypertension, and hyperlipidemia treatments.
2. `TextLoader` loads the file into LangChain `Document` objects.
3. `RecursiveCharacterTextSplitter` splits documents into chunks of 300 characters with 50-character overlap.
4. `HuggingFaceEmbeddings` with model `sentence-transformers/all-MiniLM-L6-v2` converts chunks to dense vectors locally (no API call needed).
5. `FAISS.from_documents` builds the vector store and it is saved to `faiss_index/`.

#### Querying (`get_retriever`)
1. The FAISS index is loaded from disk with the same embedding model.
2. Returns a LangChain retriever configured to return the top 2 most similar chunks (`k=2`).

---

### LangGraph ReAct Agent (`agent.py → run_agent`)

**Libraries:** `langgraph`, `langchain-google-genai`, `langchain-core`

The agent is a ReAct (Reason + Act) agent built with `langgraph.prebuilt.create_react_agent`. It is initialized once and cached.

**LLM:** `ChatGoogleGenerativeAI` with `gemini-2.5-flash`, temperature 0.3

**Tools:**

1. `patient_profile_retrieval(query: str)`  
   Fetches the current session's patient info, raw report text (first 3000 chars), and structured analysis from the in-memory session store. Used to give the agent full context about who it is treating.

2. `medical_literature_search(query: str)`  
   Invokes the FAISS retriever to find relevant medical guideline chunks. Returns the top 2 matching document contents joined by newlines.

**System Prompt:** Instructs the agent to always retrieve the patient profile first, use Markdown formatting with headers and bullet points, explain things simply, be compassionate, and always recommend consulting a real doctor for serious concerns.

**Conversation Flow:**
1. The last 10 messages from `chat_history` are passed as context.
2. The agent reasons, optionally calls tools, and produces a final response.
3. The response is extracted from the last message in the result.
4. Both the user message and AI response are appended to `chat_history`.

---

### ML Pipeline (`ml_pipeline.py`)

**Libraries:** `pandas`, `numpy`, `scikit-learn`, `seaborn`, `matplotlib`, `joblib`

This is a one-time training script (run directly, not imported by the server at runtime).

1. Generates 1000 synthetic patient records with features: `Age`, `Blood_Pressure`, `Cholesterol`, `BMI`, `Glucose`.
2. Introduces 5% random NaN values per column, then fills with column medians.
3. Computes a `risk_score` using threshold-based rules and creates a binary `Diabetes_Risk` target (1 if score > 3.0).
4. Plots a correlation heatmap with `seaborn` and saves it as `correlation.png`.
5. Splits data 80/20, scales with `StandardScaler`, trains a `LogisticRegression` model.
6. Serializes the model and scaler with `joblib` to `risk_model.pkl` and `scaler.pkl`.

The `/patient/predict` endpoint uses these artifacts for legacy diabetes risk prediction.

## Frontend

### Framework: React 19 + Vite 8

The entire UI lives in a single `App.jsx` file. `main.jsx` mounts the app into `#root` using `createRoot`.

### Libraries

| Library | Version | Purpose |
|---------|---------|---------|
| `react` | ^19.2.4 | UI framework |
| `react-dom` | ^19.2.4 | DOM rendering |
| `react-markdown` | ^10.1.0 | Renders Markdown from AI chat responses |
| `vite` | ^8.0.1 | Dev server and bundler |
| `@vitejs/plugin-react` | ^6.0.1 | Vite plugin for React/JSX transform |

### Component Architecture

All components are defined in `App.jsx` and communicate via props/callbacks. State is lifted to the root `App` component.

```
App (root state: step, sessionId, patientInfo, extractedData, analysis)
├── StepBar          — visual progress indicator (steps 1–4)
├── PatientIntake    — step 1: registration form, live BMI calculator
├── UploadReport     — step 2: drag-and-drop file upload, calls /report/upload then /report/analyze
└── AnalysisDashboard — steps 3+4: tabbed analysis panel + AI chat
```

#### `StepBar`
Renders 4 labeled steps with connecting lines. Active/completed states are driven by the `current` prop.

#### `PatientIntake`
- Controlled form with fields: name, age, gender, height, weight.
- Computes BMI live: `weight / (height/100)²` and shows a color-coded category (Underweight/Normal/Overweight/Obese).
- On submit, POSTs to `/patient/register` and calls `onComplete(sessionId, patientInfo)` to advance to step 2.

#### `UploadReport`
- Supports drag-and-drop and click-to-browse for `.pdf`, `.png`, `.jpg`, `.jpeg`, `.webp`.
- On upload click:
  1. POSTs file + session_id as `multipart/form-data` to `/report/upload`.
  2. On success, POSTs session_id to `/report/analyze`.
  3. Calls `onComplete(extractedData, analysis)` to advance to step 3.

#### `AnalysisDashboard`
Split into two panels:

**Left — Analysis Panel:**
- Shows patient badge (name, age, gender, BMI).
- Summary card with the Gemini-generated overview.
- 5 tabs: Overview, Good Findings, Concerns, Risks, Follow-up Tests.
  - Overview: stat cards (counts) + lifestyle tips + correlation heatmap image from `/static/correlation.png`.
  - Good/Concerns/Risks/Tests: mapped from the analysis JSON with color-coded severity/urgency badges.

**Right — AI Doctor Chat:**
- Displays chat messages with `ReactMarkdown` for rich formatting of AI responses.
- Quick-question buttons shown until the first user message.
- Input form POSTs to `/chat` with `session_id` and `message`.
- Auto-scrolls to the latest message via `useRef` + `useEffect`.

## End-to-End Data Flow

```
User fills form
    → POST /patient/register
    → Session created, BMI computed, session_id returned

User uploads report
    → POST /report/upload (multipart)
    → Gemini Vision reads image/PDF → returns JSON of all lab parameters
    → Stored in session["report_text"]

    → POST /report/analyze
    → Gemini analyzes patient profile + lab JSON
    → Returns structured analysis (good/bad findings, risks, tips)
    → Stored in session["report_analysis"]

User chats
    → POST /chat
    → LangGraph ReAct agent invoked
    → Agent calls patient_profile_retrieval → gets session data
    → Agent optionally calls medical_literature_search → FAISS retrieval
    → Agent generates Markdown response
    → Response rendered via ReactMarkdown in chat UI
```

## Running Locally

### Backend
```bash
cd backend
pip install -r requirements.txt
# One-time setup (if pkl/faiss files don't exist):
python ml_pipeline.py
python rag_pipeline.py
# Start server:
uvicorn main:app --reload --port 8000
```

### Frontend
```bash
cd frontend
npm install
npm run dev
```

The frontend runs on `http://localhost:5173` and calls the backend at `http://localhost:8000` by default.

## Free Deployment

### Backend → Render (free tier)
1. Push this repo to GitHub.
2. Go to [render.com](https://render.com) → New Web Service → connect your repo.
3. Set root directory to `backend`, or use the included `render.yaml` for auto-config.
4. Add environment variable: `GOOGLE_API_KEY=<your key>`.
5. Deploy. Render will run `uvicorn main:app --host 0.0.0.0 --port $PORT`.

### Frontend → Vercel or Netlify (free tier)
1. Go to [vercel.com](https://vercel.com) or [netlify.com](https://netlify.com) → import your GitHub repo.
2. Set root directory to `frontend`.
3. Add environment variable: `VITE_API_URL=https://your-backend.onrender.com`
4. Build command: `npm run build`, output directory: `dist`.
5. Deploy.

> The API URL is read from `import.meta.env.VITE_API_URL` at build time, falling back to `http://localhost:8000` for local dev.
